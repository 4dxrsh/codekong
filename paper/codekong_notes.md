# CodeKong — running notes toward a paper draft

These are working notes, not a submission.

## Hypothesis

Retrieved context from the actual codebase under test (RAG) produces test
rewrites that kill more surviving mutants than a closed-book baseline that
sees only the mutant diff and the original test (NO_RAG, mirroring MuTAP),
and this benefit grows with mutation-class difficulty: syntactic <
statement-deletion / semantic < higher-order.

## Research questions

RQ1. Does RAG beat NO_RAG on overall kill rate?
RQ2. Does RAG's benefit scale with class hardness (syntactic < semantic <
     higher_order)? Note: results are reported either way. An inconclusive or
     reversed gradient is a finding, not a failure, and must not be shaded
     into a success.
RQ3. How does retrieval depth k (3/5/8) affect valid-test rate?
RQ4. Is the extra retrieval/generation cost worth it per mutation class?
     Cost is measured from the Ollama call log (wall time, eval_count /
     prompt_eval_count token counts) since a local model has no billing
     meter.

## Method summary

Four mutant sources normalized to one schema: mutmut (syntactic, mechanical
operator swaps), Python-ast statement deletion (sdl), LLM-generated realistic
bugs (semantic, temperature 0.7), and a same-function pair combiner
(higher_order, capped at 3/function). Survivors of the subject's existing
suite enter two generation conditions (NO_RAG / RAG) through the same
class-specific prompt templates at temperature 0, with an agentic refinement
layer allowing exactly one retry driven by structured validator feedback
(pass-on-original AND fail-on-mutant required for a kill).

Subjects: TheAlgorithms/Python (sorts/ only) and dbader/schedule, both MIT,
pinned to fixed commits in config.yaml. A planned third repo (jFilho/calcpy)
was dropped after verification showed it does not exist on GitHub; two
subjects are within the stated 2–3 scope.

## Threats to validity

1. Model substitution (IMPORTANT). The original design called for the Claude
   API for both semantic-mutant generation and test rewriting. This
   implementation deliberately substitutes a local open model served by
   Ollama (default qwen2.5-coder:7b). This changes the ceiling on
   instruction-following and code-generation quality. The RAG-vs-NO_RAG
   comparison within a run remains internally valid — both conditions use
   the same model, prompts, and temperature — but absolute kill rates and
   valid-test rates are NOT comparable to a Claude-backed run and must not
   be presented as such.
2. Hardware-driven model tier. On the target laptop (RTX 3050 Ti, 4GB VRAM)
   the 7B model partially offloads to system RAM; slower generation may
   tempt shorter runs. Mutant caps are configured, not ad hoc, to keep the
   two conditions symmetric.
3. Semantic mutants are generated by the same model family that rewrites
   tests; a shared blind spot could depress semantic-class kill rates in
   both conditions. Mitigation: temperature separation (0.7 vs 0.0) and
   ast-parse + suite-survival filtering of generated mutants.
4. Subject scale. sorts/ is algorithmic and doctest-tested; schedule is a
   small library with a conventional pytest suite. Two subjects limit
   external validity; results are reported per subject.
5. Equivalent mutants. Some survivors (especially SDL on dead code) may be
   semantically equivalent to the original; neither condition can kill
   them, deflating both kill rates equally.
6. HOM construction is textual (changed-region reapplication) and skips
   overlapping pairs, biasing HOMs toward spatially separated component
   mutations.
